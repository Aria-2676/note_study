# 任务管家项目开发规范
文档版本：v1.0
更新时间：2026-04-14
适用范围：任务管家Flutter项目所有开发人员

## 一、文档说明
本规范定义了「任务管家」项目的核心开发准则，覆盖状态管理、统计系统、工程结构、命名编码等维度，旨在保障多人协作效率、降低维护成本、统一技术实现标准。所有新增/迭代功能必须遵循本规范，例外场景需经团队评审并在文档中登记。

## 二、核心架构规范
### 2.1 整体架构原则
1. 项目基于Flutter + Provider实现状态管理，核心业务逻辑（任务、积分、商城、标签）通过独立Provider封装，禁止跨Provider直接耦合；
2. 基础服务（如WidgetService、StatisticService）需在`main.dart`中初始化，且遵循「基础服务优先、业务Provider后置」的初始化顺序；
3. 所有UI层仅通过`Provider.of`/`Consumer`获取业务数据，禁止直接实例化Provider或修改其内部状态。

### 2.2 Provider状态管理规范（强制）
#### 2.2.1 Provider注册规则
1. 所有Provider需在`MultiProvider`中集中注册（参考`main.dart`），禁止在页面内零散注册；
2. 基础Provider（无依赖）：使用`ChangeNotifierProvider`注册（如AppStateProvider、SettingsProvider）；
3. 依赖型Provider（如ShopProvider/TaskProvider依赖PointsProvider）：
   - 必须使用`ChangeNotifierProxyProvider`注册；
   - `create`方法中通过`context.read<依赖Provider>()`初始化；
   - `update`方法中更新依赖引用，且需判空（`shopProvider!`）；
4. Provider注册顺序：无依赖Provider在前，依赖型Provider在后（参考`main.dart`中PointsProvider → ShopProvider/TaskProvider的顺序）。

#### 2.2.2 Provider使用规则
1. 获取Provider实例：
   - 仅读取数据（不监听变更）：`Provider.of<T>(context, listen: false)`；
   - 监听数据变更（需刷新UI）：`Provider.of<T>(context)` 或 `Consumer<T>`；
   - 禁止在非Widget层（如Adapter、Service）使用`listen: true`，避免内存泄漏；
2. 状态更新限制：
   - 仅业务逻辑变更时调用`notifyListeners()`，禁止在统计、日志等非核心逻辑中触发；
   - ProxyProvider的`update`方法仅更新依赖引用，禁止在其中执行业务逻辑；
3. Provider职责边界：
   | Provider          | 核心职责                          | 禁止处理的逻辑              |
   |-------------------|-----------------------------------|-----------------------------|
   | AppStateProvider  | 全局应用状态（如登录态、版本信息）| 业务数据（任务/积分）处理   |
   | SettingsProvider  | 应用设置（主题、语言）| 统计上报、第三方服务调用    |
   | PointsProvider    | 积分增减、积分查询                | 商城兑换、任务完成逻辑      |
   | ShopProvider      | 商城商品、兑换逻辑                | 积分计算、任务状态变更      |
   | TaskProvider      | 任务增删改查、任务完成            | 积分发放、商城商品管理      |
   | TagProvider       | 任务标签管理                      | 任务逻辑、积分相关操作      |

## 三、统计系统开发规范（核心）
### 3.1 统计系统架构原则（强制）
1. 统计系统采用「统一Service + 业务适配器」模式，核心类为`StatisticService`；
2. 所有统计操作必须通过`StatisticService.report()`上报，禁止：
   - 为单个模块（Task/Shop/Tag等）编写独立统计接口；
   - 直接操作SharedPreferences/后端上报接口；
3. `StatisticService.init()`必须在`main.dart`中`WidgetService.init()`之后执行（保证基础服务就绪）。

### 3.2 统计适配器规范
1. 适配器命名：`[业务模块]StatisticAdapter`（如`TaskStatisticAdapter`/`ShopStatisticAdapter`），需与对应Provider一一对应；
2. 适配器职责：封装对应模块的统计逻辑（如任务完成计数、商城兑换点击），禁止包含业务逻辑；
3. 适配器接入流程（新增模块必遵循）：
   ```
   Step1: 定义XXXStatisticAdapter，通过Provider.of获取对应Provider数据；
   Step2: 在Adapter中封装统计方法（如reportPageView、reportClick、reportCount）；
   Step3: UI层调用Adapter方法（禁止直接调用StatisticService）；
   Step4: 在本规范文档「3.3 统计项登记」中新增统计项key，避免重复；
   ```

### 3.3 统计项命名与数据格式规范（强制）
#### 3.3.1 统计类型枚举（StatisticType）扩展规则
1. 新增统计维度（如shop/tag）需团队评审，禁止随意扩展枚举值；
2. 核心枚举值约定：
   - `StatisticType.pageView`：页面访问埋点；
   - `StatisticType.click`：点击行为埋点；
   - `StatisticType.count`：业务计数埋点；
   - `StatisticType.system`：系统异常/基础信息埋点。

#### 3.3.2 统计项key命名规则
| 埋点类型   | 命名格式                | 示例                          |
|------------|-------------------------|-------------------------------|
| 页面访问   | page_view_[模块]_[页面] | page_view_task_home（任务首页）、page_view_shop_exchange（商城兑换页） |
| 点击行为   | click_[模块]_[动作]     | click_task_complete（完成任务）、click_shop_exchange（兑换商品） |
| 业务计数   | count_[模块]_[指标]     | count_task_completed（完成任务数）、count_points_increase（积分增加数） |
| 系统异常   | system_[类型]           | system_app_crash（App崩溃）、system_init_fail（初始化失败） |

#### 3.3.3 统计数据格式约束
| 字段         | 类型       | 约束说明                          |
|--------------|------------|-----------------------------------|
| StatisticData.key | String     | 必须遵循上述命名规则，禁止重复    |
| StatisticData.value | dynamic    | 计数用int、行为埋点用Map（含taskId/shopId等）、系统数据用String |
| StatisticData.time | DateTime   | 必须为事件发生的UTC时间，禁止手动篡改 |

#### 3.3.4 统计项登记（新增需补充）
| 统计项key                | 所属模块 | 类型       | 说明                     |
|--------------------------|----------|------------|--------------------------|
| page_view_task_home      | Task     | pageView   | 任务首页访问             |
| click_task_complete      | Task     | click      | 任务完成按钮点击         |
| count_task_completed     | Task     | count      | 累计完成任务数           |
| page_view_shop_home      | Shop     | pageView   | 商城首页访问             |
| click_shop_exchange      | Shop     | click      | 商品兑换按钮点击         |
| count_points_increase    | Points   | count      | 积分增加总数             |
| system_app_crash         | System   | system     | App崩溃信息              |

### 3.4 统计上报与存储规则（强制）
#### 3.4.1 上报时机
| 埋点类型   | 上报时机                          | 注意事项                          |
|------------|-----------------------------------|-----------------------------------|
| 页面访问   | 页面`initState`方法中             | 禁止在`build`方法中上报（避免重复） |
| 点击行为   | 按钮`onPressed`异步执行           | 禁止阻塞点击响应（如await上报后再执行业务） |
| 业务计数   | 数据变更后（如任务完成、积分增减） | 保证数据已持久化后再上报          |
| 系统异常   | `FlutterError.onError`捕获后      | 需包含异常堆栈、设备信息          |

#### 3.4.2 存储与上报限制
1. 本地存储：最多缓存2000条统计数据，超出自动清理最早数据；
2. 服务端上报：
   - 仅在release模式下执行，debug模式禁止上报；
   - 上报失败后缓存数据，下次App启动时重试；
   - 上报接口地址配置在`config/env.dart`，禁止硬编码在StatisticService中。

### 3.5 与Provider的对接规则（强制）
1. 适配器获取Provider数据：必须通过`Provider.of<T>(context, listen: false)`，禁止直接持有Provider实例；
2. 统计逻辑限制：禁止在统计适配器中调用Provider的`notifyListeners()`；
3. 依赖型Provider统计：Adapter需先获取最新依赖Provider数据（如ShopAdapter先读PointsProvider），再计算统计值；
4. 禁止统计逻辑触发UI重建：所有统计操作需与UI刷新解耦。

## 四、工程结构规范
### 4.1 核心目录结构
```
lib/
├── core/                # 核心基础层
│   ├── services/        # 全局服务（WidgetService、StatisticService）
│   ├── config/          # 配置文件（环境、常量）
│   ├── utils/           # 工具类（通用方法、扩展函数）
│   └── models/          # 全局数据模型（StatisticData、BaseModel）
├── providers/           # Provider层（所有状态管理类）
├── modules/             # 业务模块层
│   ├── tasks/           # 任务模块
│   │   ├── pages/       # 页面
│   │   ├── adapters/    # 统计适配器（TaskStatisticAdapter）
│   │   └── utils/       # 模块内工具类
│   ├── shop/            # 商城模块（同tasks结构）
│   ├── points/          # 积分模块（同tasks结构）
│   └── tag/             # 标签模块（同tasks结构）
├── docs/                # 文档（本规范、统计项登记）
└── main.dart            # 入口文件
```

### 4.2 目录/文件命名规则
1. 目录名：小写+下划线（如`tasks_home`、`statistic_service`）；
2. 文件名：
   - 所有文件：小写+下划线（如`task_provider.dart`、`task_statistic_adapter.dart`），遵循Dart官方风格指南；
   - 类名：大驼峰（如`TaskProvider`、`TaskStatisticAdapter`）；
3. 禁止中文/特殊字符命名，禁止无意义命名（如`utils1.dart`、`page.dart`）。

## 五、编码规范
### 5.1 代码风格
1. 遵循Dart官方风格指南，使用`dart format`格式化代码；
2. 缩进：4个空格，禁止使用tab；
3. 行长度：单行代码不超过120个字符，超出需换行；
4. 注释：
   - 类/方法：必须添加文档注释（///），说明功能、参数、返回值；
   - 复杂逻辑：添加行内注释，说明设计思路；
   - 禁止注释无用代码，禁止中文拼音注释。

### 5.2 异常处理
1. 所有异步操作（如网络请求、文件读写）必须捕获异常；
2. 统计系统异常：需上报至`system_app_crash`/`system_init_fail`，禁止吞异常；
3. Provider操作异常：需捕获并记录日志，禁止导致UI崩溃；
4. 示例：
```dart
// 正确示例：统计上报异常处理
Future<void> reportPageView(String key) async {
  try {
    await _statisticService.report(StatisticData(
      key: key,
      type: StatisticType.pageView,
      value: DateTime.now().toString(),
    ));
  } catch (e, stack) {
    // 记录日志+上报系统异常
    debugPrint('页面访问统计上报失败：$e\n$stack');
    await _statisticService.report(StatisticData(
      key: 'system_statistic_fail',
      type: StatisticType.system,
      value: {'error': e.toString(), 'stack': stack.toString()},
    ));
  }
}
```

## 六、扩展与迭代规则
### 6.1 规范迭代
1. 每季度评审一次本规范，根据新增业务场景（如新增支付模块）补充规则；
2. 规范变更需团队一致同意，变更后同步更新`docs/`下的文档，并在代码中更新注释引用。

### 6.2 临时需求处理
1. 临时统计/业务需求：需在`docs/temp_requirements.md`中登记，包含「需求内容、上线时间、清理时间」；
2. 临时代码需添加`// TEMP: 清理时间-责任人`注释，到期必须清理，禁止长期保留。

### 6.3 兼容性保障
1. `StatisticService`/Provider的对外接口（如`report`、`getByType`）需保持向后兼容；
2. 底层逻辑优化（如存储方式、上报接口）需保证不影响上层调用，必要时做兼容适配。

## 七、附则
1. 本规范自发布之日起生效，所有新增代码需严格遵循；
2. 历史代码迭代时，需逐步重构至符合本规范；
3. 违反本规范导致的问题（如数据混乱、UI崩溃），由开发人员承担相应责任；
4. 本规范最终解释权归「任务管家」开发团队所有。
